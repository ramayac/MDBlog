# MDBlog — Agent Instructions

## Project Overview

MDBlog is a lightweight, flat-file blog engine written in **Go 1.24**. Posts are Markdown files with YAML-style front matter. There is no database, no build step, and no JavaScript framework. Keep it that way.

## Architecture

```
cmd/
  mdblog/           # CLI entry point: serve | build-index | build-feed | render | version
  lambda/           # AWS Lambda entry point (reads templates/assets from disk)
  lambda-embed/     # AWS Lambda entry point (templates+assets embedded in binary)
internal/
  blog/             # Core domain: post scanning, slug resolution, pagination, menu
  buildindex/       # Build-time index generator (writes posts/posts.index.json)
  buildfeed/        # Build-time RSS feed generator (writes feed.xml from the index)
  config/           # Config loader — parses config.toml into Config struct
  markdown/         # Front matter parser + Goldmark renderer
  render/           # CLI render subcommand (renders a post to a standalone HTML file)
  server/           # net/http handler: routing, template execution, gzip, CSP
templates/          # Go html/template files (*.html)
assets/
  css/              # CSS themes
  js/               # Optional per-post JavaScript files
posts/              # All content lives here
  index.md          # Optional landing page blurb (shown above category cards)
  *.md              # Root-level (uncategorized) posts
  srbyte/           # Category folder
  substack/         # Category folder
posts/posts.index.json  # Pre-built post metadata index (generated by `make build-index`)
feed.xml            # Pre-built RSS 2.0 feed (generated by `make build-feed`, baked into Docker image)
embed.go            # go:embed declarations; exposes EmbeddedTemplates() / EmbeddedAssets()
config.toml         # All runtime configuration (single source of truth)
Dockerfile          # Standard Lambda image: builds lambda binary, copies posts/templates/assets
Dockerfile.embed    # Embed variant: binary only + posts + config.toml
docker-compose.yml  # Local dev only — runs /mdblog serve on port 8080
Makefile            # Developer targets: help, serve, build, test, new-post, docker-*
```

**Deployment model:** The production artifact is a Docker container image pushed to a registry (e.g. ghcr.io) and deployed as an **AWS Lambda container image function** behind API Gateway. The `cmd/lambda` binary uses [algnhsa](https://github.com/akrylysov/algnhsa) to translate API Gateway HTTP events into standard `http.Handler` calls. Posts and all static assets are **baked into the image** — there is no external content source or database.

**Embed variant:** `cmd/lambda-embed` builds a single binary that has `templates/` and `assets/` embedded via `go:embed`. The resulting image only requires the binary, `posts/`, and `config.toml` — useful when you want the smallest possible image footprint.

**Continuous Deployment:** A GitHub Action is configured to automatically build and release a new container image to GHCR whenever a Markdown (`.md`) file inside the `posts/` folder is pushed to `master`. A chained workflow then automatically deploys the latest image to AWS ECR and updates the Lambda function.

`docker-compose.yml` is for local development only and is not used in production.

## Configuration

All settings live in `config.toml` (TOML format). They are parsed by `internal/config` into a `Config` struct. Key fields:

| Key | Type | Purpose |
|-----|------|---------|
| `blog_name` | string | Site title |
| `author_name` | string | Default post author (used by `make new-post`) |
| `lang` | string | HTML `lang` attribute (default `"en"`) |
| `header_content` | string | Landing page subtitle |
| `footer_content` | string | Footer text (Markdown supported) |
| `posts_per_page` | int | Pagination size (default 10) |
| `excerpt_length` | int | Max excerpt length in characters (default 200) |
| `show_uncategorized` | bool | Include root-level posts in listings |
| `uncategorized_label` | string | Display name for uncategorized posts |
| `posts_dir` | string | Root posts directory (default `"posts"`) |
| `post_index_file` | string | Path to pre-built metadata index |
| `date_format` | string | Go time layout for dates (default `"2006-01-02"`) |
| `css_theme` | string | Path to active CSS theme |
| `show_render_time` | bool | Append render-time HTML comment |
| `[csp]` | struct | `enabled` + `header` string |
| `[feed]` | struct | `enabled`, `title`, `description`, `base_url` (req when enabled), `max_items` (default 50), `output_file` (default `"feed.xml"`) |
| `[[menu_links]]` | array | Static nav links: `label` + `url` |
| `[categories.<slug>]` | map | Category definitions (see below) |
| `[labels]` | struct | All user-visible UI strings |

When adding a new config key, add it to `config.toml` and the `Config` struct in `internal/config/config.go`, then consume it via the `*config.Config` pointer passed to handlers. Never hardcode values in templates or handlers.

## Post Format

```yaml
---
title: My Post Title
date: YYYY-MM-DD
author: Name
tags: tag1, tag2
description: Optional meta description
js: optional-script.js   # loaded from assets/js/
---

Markdown body here (GitHub Flavoured Markdown + footnotes).
```

File naming convention: `YYYY-MM-DD-slug-with-hyphens.md`
Category posts live under `posts/<category-folder>/`.

## Developer Workflow

```bash
make serve                                                    # Start local HTTP server at localhost:8080
make build                                                    # Compile production binaries to bin/
make build-embed                                              # Compile embed-variant binary to bin/lambda-embed
make build-index                                              # Generate posts/posts.index.json
make build-feed                                               # Generate feed.xml (requires build-index first)
make lint                                                     # Run go vet on all packages
make lint-config                                              # Parse and validate config.toml
make test                                                     # Build index + feed then run go test ./...
make render random                                            # Render a random post to HTML
make render [category] random                                 # Render a random post from a category
make render filename.md                                       # Render a specific post to HTML
make new-post TITLE="Title" [CATEGORY=slug] [TAGS="t1, t2"]  # Scaffold a new post
```

`make new-post` reads `author_name` from `config.toml` automatically.

**Re-run `make build-index` after adding or editing posts** so that listing and pagination pages reflect changes immediately. Without the index, the blog falls back to a live filesystem scan (correct but slower).

## Docker Workflow

```bash
make docker-build                               # Build production Docker image (FROM scratch, Lambda-ready)
make docker-build-embed                         # Build embed-variant Docker image
make docker-run                                 # Run via docker compose (localhost:8080)
make docker-run-release                         # Run pulled release image without rebuilding
make docker-stop                                # Stop and remove Docker Compose containers
make docker-push                                # Tag and push to ghcr.io/ramayac/mdblog
make docker-pull [TAG=1.2.3]                    # Pull release image and retag as mdblog:latest
```

`make docker-build` compiles the Go binaries and generates the post index inside the Docker build stage. The index is baked into the final image and is present at Lambda runtime without any filesystem writes.

The standard `Dockerfile` uses a **multi-stage build**: a `golang:1.24` build stage compiles the binaries and generates the index, then copies only the necessary artifacts into a minimal `FROM scratch` production image. The container runs with a read-only filesystem, all Linux capabilities dropped, and no new privileges.

**When modifying the Makefile**, always update `README.md` and `AGENTS.md` to reflect the new or changed targets.

## Internal Packages

### `internal/config`

Loads and validates `config.toml`. Exported functions:
- `Load(path string) (*Config, error)` — parse and apply defaults
- `MustLoad(path string) *Config` — panics on error (used by entry points)

### `internal/markdown`

Front matter extractor and Goldmark-based renderer.
- `Parse(content string) ParsedDoc` — extract front matter + render body to HTML
- `ParseMetaOnly(content string) MetaDoc` — extract front matter + return raw body (no rendering; used by `buildindex`)
- Goldmark is configured with GFM extensions (tables, strikethrough, task lists), footnotes, auto heading IDs, hard wraps, and **safe mode** (no raw HTML passthrough — do not add `html.WithUnsafe()`).

### `internal/buildindex`

Scans all post directories, calls `ParseMetaOnly` for each, and writes `posts/posts.index.json` atomically. Full Goldmark rendering is **never invoked** during index generation.
- `Build(cfg *config.Config) error`

### `internal/blog`

Core domain logic. The `Blog` struct is the main service object:
- `New(cfg *config.Config) *Blog`
- `GetPosts(category string, page int) (PostList, error)` — paginates from the JSON index
- `SearchPosts(query string, page int) (PostList, error)` — full-text search via the JSON index
- `GetPostBySlug(slug, category string) (*Post, error)` — reads + renders a single post
- `GetCategories() map[string]*CategoryInfo`
- `GetCategoryBySlug(slug string) (*CategoryInfo, bool)`
- `GetMenu() []MenuLink`
- `GetVersionInfo() VersionInfo`

### `internal/server`

`net/http` handler. Key details:
- `New(cfg *config.Config, b *blog.Blog) *Handler` — loads templates, returns the handler
- `TemplateFS` and `AssetsFS` package-level variables — override in embed builds
- Routes: `/` (home/category/search), `/feed.xml` (RSS), `/feed` (human-readable feed page), `/post` (single post), `/assets/*` (static files)
- Gzip compression via `compress/gzip` (when `Accept-Encoding: gzip` is present)
- CSP header injected per request when `csp.enabled = true`
- Templates use `html/template` with custom functions: `safeHTML`, `formatDate`, `urlquery`, `postPreviewData`, `defaultStr`

### `internal/buildfeed`

Build-time RSS 2.0 feed generator.
- `Build(cfg *config.Config) error` — reads `posts.index.json`, writes `feed.xml` atomically
- No Goldmark rendering — excerpts come from the pre-built index
- No external dependencies — uses `encoding/xml` from the standard library
- Called by `make build-feed`; must be run after `make build-index`
- `feed.enabled = true` in `config.toml` requires `feed.base_url` to be set (returns error otherwise)

### `internal/render`

CLI subcommand for rendering a single post to a standalone HTML file in a `render/` output directory. Used by `make render`. Performs a fake HTTP round-trip through the `server.Handler` to reuse all template logic.

## Navigation Menu

The nav bar is built by `blog.GetMenu()`. Two sources, rendered in order:
1. `[[menu_links]]` entries in `config.toml` — static custom links.
2. Any category with `menu = true` in its config block — auto-generates a `?category=slug` link.

## Category Config Flags

Each `[categories.<slug>]` block in `config.toml` supports:

| Flag | Type | Effect |
|------|------|--------|
| `blog_name` | string | Display name in nav and category header |
| `header_content` | string | Subtitle shown on category listing page |
| `folder` | string | Subfolder under `posts/` (defaults to slug) |
| `index` | bool | Include in legacy aggregated listing |
| `menu` | bool | Add link to the nav bar |

## Landing Page, Category Page, Search, and Feed

The server handler has **five rendering paths**:

- **`/` with no query params** → Static landing page. Shows the blog header, optional `posts/index.md` blurb, then auto-generated category cards. **No post scanning.**
- **`/?category=slug`** → Category post listing with pagination.
- **`/?q=keyword`** or **`/?search=1`** → Standalone search results page using the JSON index.
- **`/feed.xml`** → Serves the pre-built RSS 2.0 feed from disk. Returns 404 when `feed.enabled = false`.
- **`/feed`** → Human-readable feed page listing all posts (date | category | title). Uses `blog.GetFeedPosts(maxItems)`.

To add a landing page blurb: create `posts/index.md` with Markdown content. Delete it to show only the category cards.

## Coding Conventions

- **Go 1.24+ required.** Use standard Go idioms; avoid unnecessary abstractions.
- **No ORMs, no frameworks** — standard library `net/http` + Goldmark + go-toml are the only non-AWS dependencies.
- **XSS prevention:** post content is rendered by Goldmark **without** `html.WithUnsafe()`. Template variables in `html/template` are auto-escaped. Raw HTML is only emitted via explicit `template.HTML` casts in trusted code paths.
- **Path traversal prevention:** `GetPostBySlug` validates that resolved file paths are within `PostsDir`. Any new file-reading code must do the same.
- **CSP:** configured via `config.toml` `[csp]` block. Do not add `unsafe-eval` without a documented security justification.
- **Gzip:** the server handler applies gzip compression. Any new HTTP entry point should do the same.
- **No globals:** pass `*config.Config` and `*blog.Blog` explicitly. Do not introduce package-level mutable state (except the `TemplateFS`/`AssetsFS` embed overrides).
- **Tests live in the same package** (white-box style). Run with `make test`.

## Adding a New Category

1. Create `posts/<folder-name>/` and add `.md` files.
2. Register the category in `config.toml`:
   ```toml
   [categories.my-category]
   blog_name      = "Display Name"
   header_content = "Subtitle shown on category index."
   folder         = "my-category"
   index          = false
   menu           = true
   ```
3. Run `make build-index` to regenerate the post metadata index.

## Post Metadata Index

Listing, pagination, and search are driven by a **build-time metadata index** (`posts/posts.index.json`). This avoids scanning and parsing all Markdown files on every request.

- **Generated by:** `make build-index` → `internal/buildindex.Build()`
- **Used by:** `blog.GetPosts()` (listing + pagination), `blog.SearchPosts()` (search), and `blog.GetPostBySlug()` (fallback slug→category resolution)
- **Not used for rendering:** individual post pages still parse the full Markdown body, but only for the single requested post
- **Fallback:** if the index is absent or corrupt, `GetPosts()` falls back to a live filesystem scan and logs a warning

### Index contents (per post)

| Field | Source |
|-------|--------|
| `slug` | derived from filename |
| `title` | front matter or filename |
| `date` | front matter or file mtime |
| `author` | front matter |
| `tags` | front matter |
| `description` | front matter |
| `excerpt` | `description` or plain-text strip of raw Markdown body |
| `category_slug` | parent folder slug |
| `source_path` | relative path to `.md` file |
| `filename` | basename of the `.md` file |

Full post bodies are **never rendered** during index generation (Goldmark is not called).

## Lambda Deployment Notes

- The Lambda binary is a statically linked Go binary (`CGO_ENABLED=0`). No runtime dependencies.
- `algnhsa` adapts API Gateway v1/v2 HTTP events into `http.Handler` calls transparently.
- The container filesystem is read-only. There is no file-based cache — the JSON index is baked into the image and read-only at runtime.
- **Recommended caching strategy:** place **CloudFront** in front of the Lambda function. Since posts change only on redeploy, a CloudFront TTL of hours or days is safe. Invalidate the distribution after `make docker-push`.

## What NOT to Do

- Do not add a database layer.
- Do not introduce a front-end build tool (webpack, vite, etc.).
- Do not set `html.WithUnsafe()` in the Goldmark configuration.
- Do not add `unsafe-eval` to the CSP without a documented security justification.
- Do not put new content in the root `posts/` dir (unless `show_uncategorized = true`) — use a category folder.
- Do not run the container as root or re-add dropped Linux capabilities without justification.
- **When modifying the Makefile**, always update `README.md` and `AGENTS.md` to match.
