Go, Lambda, and Docker: Running a flat-file blog at near-zero cost

2026-03-29 in Personal 🏠 By Rodrigo Amaya

When I decided to build my own blog engine, I had one hard constraint: it should cost essentially nothing to run. The combination of Go, AWS Lambda, and Docker makes that possible — and the architecture is simple enough to understand in an afternoon.

Why Go?

Go compiles to a single statically-linked binary with no runtime dependencies. That's exactly what you want for a Lambda container image. Cold starts are fast, the binary is small, and the standard library covers everything a simple blog needs: net/http, html/template, file I/O.

The multi-stage Dockerfile

The key insight is that posts are baked into the container at build time. There's no database, no S3 bucket, no CMS API to call at runtime. The Lambda function is effectively a read-only static site server with Markdown rendering on the fly. 

FROM golang:1.24 AS build
WORKDIR /app
COPY . .
RUN make build-index && CGO_ENABLED=0 go build -o /bin/lambda ./cmd/lambda

FROM scratch
COPY --from=build /bin/lambda /lambda
COPY posts /posts
COPY templates /templates
COPY assets /assets
COPY config.toml /config.toml
ENTRYPOINT ["/lambda"]

CloudFront as the caching layer

Since posts only change on deploy, CloudFront with a long TTL (hours or days) means the Lambda function barely runs. Most requests are served entirely from the CDN edge.

The best infrastructure is the infrastructure you never have to think about.

The whole thing — domain, CDN, Lambda invocations — costs less than a cup of coffee per month for a low-traffic blog. And when traffic spikes (thanks, Hacker News), Lambda scales automatically.

  • No database to manage or back up
  • No server to patch
  • No build step beyond make docker-push
  • Content is just Markdown files in a Git repository
